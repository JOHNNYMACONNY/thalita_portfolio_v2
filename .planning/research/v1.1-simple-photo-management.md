# Research: v1.1 Simple Photo Management

**Date:** 2026-04-16
**Milestone:** v1.1 Simple Photo Management

## Goal

Identify interface patterns that make photo uploading, album/category assignment, and image placement feel obvious for a non-technical editor.

## Best Reference Patterns

### 1. Apple Photos / iCloud Photos

Sources:
- https://support.apple.com/en-lamr/guide/icloud/mm6feb05b125/icloud
- https://support.apple.com/en-la/102235

Useful patterns:
- Albums are easy to create, rename, and drag items into.
- Removing an item from an album does not imply deleting the original asset.
- Sidebar album structure is understandable even for non-technical users.

Why it matters here:
- Thalita should never feel like assigning an image to a category is risky or destructive.
- "Category" should behave mentally like "album" rather than like a CMS relationship field.

### 2. Pixieset Client Gallery

Sources:
- https://help.pixieset.com/hc/en-us/articles/115003740132-Uploading-to-Client-Gallery
- https://help.pixieset.com/hc/en-us/articles/41219071036685-Bulk-creating-collections-in-Client-Gallery
- https://help.pixieset.com/hc/en-us/articles/115003792812-Creating-Collections-Sets
- https://help.pixieset.com/hc/en-us/articles/43631881423885-Searching-your-collection-photos

Useful patterns:
- Batch upload is front-and-center and drag-and-drop friendly.
- Bulk creation and bulk organization reduce repetitive editor work.
- Collections, sets, and library views make it easy to see both grouped and ungrouped content.
- Search and metadata panels help users recover from messy uploads.

Why it matters here:
- This is the closest match to Thalita's real workflow: many photos, light organization, fast placement decisions.
- The presence of both a "library" view and grouped destination views suggests we should support both `Unassigned` and category slots.

### 3. Squarespace Gallery Editing

Source:
- https://support.squarespace.com/hc/en-us/articles/206543047-Gallery-pages-in-version-7-0

Useful patterns:
- Reordering happens through direct thumbnail dragging.
- Multi-select reorder is supported instead of forcing one-at-a-time changes.
- The visible edit surface mirrors the display surface closely enough that placement feels predictable.

Why it matters here:
- Reordering homepage and category images should feel visual, not numerical.
- Editors should not need to type order integers unless we keep them as an implementation detail behind the UI.

## UX Principles For This Milestone

### 1. Photo-first, not document-first

The editor should start with images and thumbnails, not with a list of `galleryItem` records.

### 2. Categories should feel like slots

The interface should make the three destinations visually obvious:
- Editorial
- Commercial
- Personal Styling

This should feel closer to "put these photos here" than "edit references on documents."

### 3. Unassigned is essential

New uploads need a clear holding area so the editor can upload first, then sort.

### 4. Bulk actions must be first-class

The workflow breaks down if the editor must open one image at a time to:
- assign category
- toggle homepage display
- hide/show images
- reorder placement

### 5. Placement state must be legible

For each image, the UI should answer:
- Which category is it in?
- Is it visible on the site?
- Is it on the homepage?
- Is it still unassigned?

### 6. Destructive actions must be emotionally safe

The UI should distinguish clearly between:
- remove from category
- hide from site
- delete image

## Recommended Product Direction

### Primary editor surface

Build a custom Studio workflow for image management rather than relying on default document editing alone.

Recommended layout:
- Upload area at the top
- `Unassigned` tray or column
- Three large category slots/cards
- Bulk action bar for selected images
- Placement badges on each thumbnail

### Ordering model

Keep underlying order fields if needed for the frontend, but expose ordering through drag-and-drop in the editor UX.

### Language choices

Use plain labels like:
- `Upload Photos`
- `Not Yet Assigned`
- `Show on Homepage`
- `Visible on Site`
- `Move to Category`

Avoid exposing raw CMS concepts like:
- slug
- reference
- document
- publish state jargon where it is not necessary

## Milestone Implications

This research supports a three-phase milestone:
1. Simplify upload and metadata entry
2. Add visual category assignment and bulk organization
3. Add clear placement/reordering controls and validate the flow with a non-technical editor
